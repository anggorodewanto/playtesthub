## Intra-milestone ordering

Per-milestone execution sequence. Each phase is red–green TDD per [`engineering.md`](engineering.md) §4 — tests land with the code, not after. The ordering minimises rework: proto and schema lock the contracts before any handler depends on them; repository tests exercise real SQL before service handlers mock repos; per-RPC vertical slices give a demoable golden flow as early as possible.

9.2. `[~]` **Self-hosted Discord login (build-our-own login page, shared-cloud-compatible)** — replaces the AGS-hosted `/auth/` redirect with a player-side flow that calls `/iam/v3/oauth/platforms/discord/authorize` directly. AGS IAM requires a `request_id` from a prior `/iam/v3/oauth/authorize` call to drive that endpoint, and the 302 from `/oauth/authorize` is an opaque redirect cross-origin (browser cannot read the `Location` header). Step 1 runs server-side via the new `Player.GetDiscordLoginUrl` proxy RPC. **Endpoint shape verified end-to-end against `abtestdewa-pong` ISC** before any code landed: `/iam/v3/oauth/authorize` 302s with `request_id`, and `/iam/v3/oauth/platforms/discord/authorize?request_id=…` 302s straight to `discord.com/api/oauth2/authorize` — no Admin Portal feature flag in the way. Code landed; manual browser smoke pending. Scope:
    - **New backend RPC `Player.GetDiscordLoginUrl`** — public/unauth, on the `Player` proto service. Request:
        ```proto
        message GetDiscordLoginUrlRequest {
          string redirect_uri          = 1; // must be on the player IAM client's allowlist
          string state                 = 2; // opaque, returned to the redirect_uri
          string code_challenge        = 3; // PKCE per RFC 7636
          string code_challenge_method = 4; // "S256"
          string scope                 = 5; // e.g. "account commerce social publishing analytics"
        }
        message GetDiscordLoginUrlResponse {
          string login_url = 1; // navigate the browser to this URL
        }
        ```
        HTTP annotation: `POST /v1/player/discord/login-url`. No `userId` in request — pre-auth.
    - **Handler**: server-side `http.Get` with `CheckRedirect = http.ErrUseLastResponse` (manual redirect handling) against `${AGS_BASE_URL}/iam/v3/oauth/authorize?response_type=code&client_id=${PLAYER_IAM_CLIENT_ID}&redirect_uri=${redirect_uri}&state=${state}&code_challenge=${code_challenge}&code_challenge_method=${code_challenge_method}&scope=${scope}`. Status MUST be 302; parse `Location` header. Reject if Location's query carries `error=…` (e.g. `error_description=redirect URI invalid` → `InvalidArgument`; `error=server_error` → `Unavailable`). Extract `request_id` from Location. Return `${AGS_BASE_URL}/iam/v3/oauth/platforms/discord/authorize?request_id=…&client_id=…&redirect_uri=…`. Response MUST NOT include the `state` or PKCE values — they remain in the player's `sessionStorage` (security: the verifier never traverses the wire).
    - **New env var `PLAYER_IAM_CLIENT_ID`** in `pkg/config` (PRD §5.9 add). The backend's confidential `AGS_IAM_CLIENT_ID` is **not** the same client; `d6bb5dd2…` is. Smoke harness must export both. Document in `docs/engineering.md` and `.env.template`.
    - **Errors `errors.md` rows**: `GetDiscordLoginUrl` failures — `InvalidArgument` (bad redirect_uri / missing PKCE / bad state shape), `Internal` (AGS IAM unreachable), `Unavailable` (AGS IAM 5xx). Byte-exact strings per the existing `errors.md` convention.
    - **Player change**: in `player/src/routes/Landing.svelte::handleSignup`, replace the local `buildDiscordLoginUrl(...)` call with `await fetchDiscordLoginUrl(config, { state, codeChallenge, redirectUri, scope })` (new helper in `src/lib/auth.ts`), then `window.location.href = response.login_url`. Drop `buildDiscordLoginUrl` and the `idp_hint=discord` query param entirely. `exchangeCodeForToken` and `Callback.svelte` are unchanged — they already handle the AGS-issued `code` correctly. Update `tests/auth.test.ts` to assert the new helper's request body + URL.
    - **Smoke harness**: extend `scripts/smoke/cloud.sh` with one POST to the new RPC asserting (a) 200, (b) response body's `login_url` contains `oauth/platforms/discord/authorize?request_id=` and the player's `client_id`. No live IAM round-trip — that's manual.
    - **Engineering doc rewrite**: replace the Discord URL-shape table in `docs/engineering.md` to describe the proxy-RPC flow honestly. Drop the now-misleading mention of `idp_hint=discord`.
    - **Verification**: manual smoke — open `http://localhost:5173/#/playtest/<seeded-slug>`, click Sign up, complete Discord auth (now on Discord's domain, not on AGS's `/auth/`), land back on `/callback`, see `/signup` render, submit, land on `/pending`, confirm the Applicant row in Postgres has `status=PENDING` and `userId` populated.

    Out of scope: NDA accept UI, code retrieval UI, survey (M2/M3). Out of scope for the player: any login surface for non-Discord platforms (covered in M3 if/when added). The backend RPC's design is generic enough to extend to other platforms later (the platform ID would just become a request field), but that's a future change.
