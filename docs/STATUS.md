## Intra-milestone ordering

Per-milestone execution sequence. Each phase is red–green TDD per [`engineering.md`](engineering.md) §4 — tests land with the code, not after. The ordering minimises rework: proto and schema lock the contracts before any handler depends on them; repository tests exercise real SQL before service handlers mock repos; per-RPC vertical slices give a demoable golden flow as early as possible.

9.3. `[x]` **Discord federation via platform-token grant (replaces 9.2 architecture)** — switch from AGS's PKCE authorization-code federation to AGS's **platform-token grant** at `POST /iam/v3/oauth/platforms/discord/token`. That endpoint accepts `platform_token=<discord-auth-code>`, exchanges it with Discord using AGS's stored Discord client secret, auto-creates the missing Justice platform account in the game namespace, and returns an AGS access token in one round trip. No `/iam/v3/oauth/authorize` hop, no `request_id`, no AGS-issued auth code, no PKCE.

    **Architecture**: player runs Discord OAuth directly (Discord developer portal is the registered redirect-URI authority, not AGS), then hands the resulting Discord code to our backend, which uses confidential AGS creds to call the platform-token grant.

    1. Player navigates to `https://discord.com/oauth2/authorize?response_type=code&client_id=${DISCORD_CLIENT_ID}&redirect_uri=${PLAYER_ORIGIN}/callback&scope=identify+email&state=${state}`.
    2. Discord redirects to `${PLAYER_ORIGIN}/callback?code=<discord-code>&state=...` (no platform_id query, since this is direct Discord OAuth — not AGS-mediated).
    3. `Callback.svelte` reads `code` + `state`, validates `state` against `sessionStorage.pendingLogin`, POSTs `code` to a new backend RPC `Player.ExchangeDiscordCode`.
    4. Backend authenticates with `AGS_IAM_CLIENT_ID`/`AGS_IAM_CLIENT_SECRET` (Basic auth, confidential client) and calls `POST /iam/v3/oauth/platforms/discord/token` with `platform_token=<discord-code>` against the configured `${AGS_BASE_URL}`.
    5. AGS returns an access token + refresh token + expires_in. Backend forwards them to the player verbatim. Player stores `access_token` in `sessionStorage` (existing `setAccessToken` helper).
