# playtesthub — M5 Build Status (scoping)

Phase tracker for **M5 — auto-approve + ADT distribution model + UX revamp**. Split into three tracks to unblock progress:

- **Track A — Auto-approve (priority 1, ships first).** An `autoApprove` option available to **every** distribution model that bypasses the manual applicant queue up to a configurable cap. Reuses the existing reserve→fenced-finalize approve path; no ADT dependency. Self-contained and shippable today.
- **Track B — ADT distribution (priority 2, ships after Track A).** A third `distributionModel` (`ADT`) that swaps the code-pool grant for an [AccelByte Development Toolkit](https://docs.accelbyte.io/development-toolkit/) build-download-link DM. Auth to ADT is via the existing playtesthub AGS service IAM JWT (no separate credential exchange / storage) — ADT validates the JWT against AGS JWKS and derives the studio identity from its `iss` / `union_namespace` claims. The live HTTP-backed adapter (`pkg/adt.HTTPClient`) ships behind an `ADT_BASE_URL` gate in bootapp; dev / smoke / e2e boots fall back to `MemClient` so the full code path keeps running offline.
- **Track C — UX revamp + admin telemetry surfaces (priority 3, ships after Track B).** A wholesale admin restructure driven by UX-supplied prototype at [`playtesthub-adt.netlify.app`](https://playtesthub-adt.netlify.app/). Replaces today's list+modal admin shell with a list+detail-page shell whose detail page is split into four tabs (*Playtest Info / Distribution / Participants / Discord Bot Tools*). Adds three new product surfaces: (i) **per-applicant ADT telemetry** (download date, total playtime, hardware specs, crash report count) sourced from ADT via cached snapshots refreshed by a worker; (ii) **bulk Discord announcements** — an admin-authored DM broadcast (subject + message) sent to Approved / Pending / All applicants via the existing PlaytestHub Discord bot, with a persistent announcement-history audit trail; (iii) **new playtest metadata** — banner image URL, description, multi-platform selector. Distribution tab adapts per model: ADT shows linkage state + build picker (B7's existing affordances reshaped); STEAM_KEYS shows code-pool upload + remaining count; AGS_CAMPAIGN shows campaign / item summary + sync affordance. **Default UX semantics for M5.A (silent PENDING fallback when cap reached) are preserved** — the prototype's "playtest is full" copy is treated as a mock inaccuracy; cap accounting stays best-effort, manual approve stays uncapped. Track C blocks on Track B because the per-applicant telemetry refresh worker calls the same ADT endpoints `pkg/adt` already wraps, and the Distribution tab's ADT panel reuses B4's linkage state machine. Track C is **all-in scope by user direction** — UX shell + new backend surfaces ship in the same milestone; the heavier ADT telemetry endpoints are flagged cut-if-behind in case ADT-eng's per-applicant telemetry surface isn't ready in time.

Conventions match [`STATUS.md`](STATUS.md) / [`STATUS_M4.md`](STATUS_M4.md) (one-line outcome per phase, pointer to the canonical doc for the *what* and *how*). This is a **scoping document** — every phase is `[ ]` not started until the corresponding PRD revision lands.

## Why M5

Two independent operator pain points motivate one milestone because they intersect at the approve path:

1. **STEAM_KEYS and AGS_CAMPAIGN both require a pre-staged code pool.** Studios that ship in-development unreleased builds via ADT have no Steam/Epic key to hand out — the deliverable is the build itself. Today playtesthub can't model that distribution.
2. **Manual approval doesn't scale for fast-iteration playtests.** A studio running a daily ADT-build playtest with hundreds of internal testers does not want to click Approve N times. They want "first N who sign up get in, rest go to PENDING for triage". The existing reserve→fenced-finalize approve path already does the right work on a per-applicant basis; auto-approve just needs to invoke it from the signup handler subject to a cap.

(2) is orthogonal to (1) — auto-approve makes sense for any model — but (1) forces a re-shape of the post-approve DM (no code value to embed; a download URL instead). The two share the same milestone because they intersect at the approve path, but the priority split (Track A → Track B) reflects that (2) can ship today against STEAM_KEYS / AGS_CAMPAIGN while (1) waits on ADT-eng's API spec.

## Design choices (resolved before A1 / B1)

### D1 — ADT linkage scope: per-studio, not per-playtest, not per-game-namespace

| Option | Summary | Verdict |
| --- | --- | --- |
| **A. Per-studio linkage** | Admin links each ADT namespace once via a state-bearing redirect to ADT; the linkage row lives on `adt_linkage` keyed `(playtesthub.studio_namespace, adt.namespace)`. Every game namespace inside the studio reuses the same linkage. `studio_namespace` is derived server-side from the **playtesthub backend's own AGS service IAM JWT** (`token.union_namespace ?? token.namespace`) — NOT from the calling admin's request token, because every downstream ADT API call from playtesthub carries the backend service token and ADT keys its side's flag on what that token represents. Studio-namespace service tokens carry `namespace` equal to the studio namespace; game-namespace service tokens always carry `union_namespace`. | **Chosen.** One link per studio covers every game and every playtest underneath. ADT side stores only a flag `(adt_namespace, studio_namespace) linked = true` — no credential is exchanged or returned. Linkage state lives separately from playtest state so re-link doesn't perturb in-flight playtests. |
| B. Per-playtest linkage | The Link ADT modal opens inside the Create-Playtest form; linkage is bound to one playtest row. | Rejected. Forces re-link on every new playtest against the same ADT namespace — direct UX regression on the actual operator workflow (daily playtests). Also splays linkage state across playtest rows where it doesn't logically belong. |
| C. Per-game-namespace linkage | Linkage keyed `(playtesthub.game_namespace, adt.namespace)`; each game under a studio links independently. | Rejected. Studios with multiple games would re-do the same operator dance once per game with no added value — operator already implicitly trusts ADT at the studio level once. Studio-scope (A) collapses N links into one without losing any safety property (ADT still validates the per-call service JWT). |
| D. Studio-global API key in env var | Operator drops `ADT_API_KEY` into Extend secrets at deploy time; no in-app linking. | Rejected. Makes the admin UI a black box ("why can't I see my ADT builds?") and forecloses on multi-namespace studios. Not needed as a fallback either — the service-JWT path is itself zero-touch once the studio is linked.

### D2 — Linking signal: full-page redirect with state token, studio_namespace query param, NO grantCode exchange

This is the question the user flagged in the scoping interview ("how does playtesthub know the link succeeded?"). Three candidates considered.

| Option | Summary | Verdict |
| --- | --- | --- |
| **A. Full-page redirect + state token + `studio_namespace` query param** | Admin clicks Proceed; admin UI persists form draft to `sessionStorage`, generates a CSRF-style `state` nonce, sets `window.location.href = ${ADT_BASE_URL}/oauth/link?state=…&redirect_uri=${ADT_REDIRECT_BASE_URL}/adt-link-callback&studio_namespace=${studio_ns}`. `studio_namespace` is **mandatory** — backend derives it server-side from the **backend's own AGS service IAM JWT** (`union_namespace ?? namespace`) — NOT the calling admin's request token, since the service JWT is what ADT validates on every downstream API call — and embeds it in the `linkUrl` it returns from `StartADTLink`; the admin UI never composes it. ADT does its sign-in + namespace select, then records the flag `(adt_namespace, studio_namespace) linked = true` on its side and redirects to `${ADT_REDIRECT_BASE_URL}/adt-link-callback?state=…&result=success&adt_namespace=<picked>` (or `result=failed&reason=…`). **`adt_namespace` is echoed on the callback URL** so playtesthub learns which namespace the operator picked without needing an extra ADT API call — `state` defends CSRF, and operator-browser tampering with `adt_namespace` is self-defeating because the first downstream ADT API call would 401 (ADT's flag wouldn't match the tampered value). **No credential, token, or `grantCode` is returned in the callback.** Admin UI callback route reads `state` + `result` + `adt_namespace` from the URL and posts `(state, adt_namespace)` to `CompleteADTLink`; backend validates the `adt_link_pending` row and inserts the `adt_linkage` row — no outbound call to ADT. Every subsequent ADT API call (e.g. `ListBuilds`, `IssueDownloadURL`) is authed by playtesthub minting an AGS service IAM JWT (existing `AGS_IAM_CLIENT_*` env vars) and sending it as `Authorization: Bearer …`; ADT validates against AGS JWKS and reads studio identity from `iss` / `union_namespace`. | **Chosen.** Robust against popup blockers; works under the Extend App UI iframe constraints; state token defeats CSRF. The "lost form draft" risk is mitigated by sessionStorage persist before redirect. Dropping the `grantCode` exchange collapses an entire phase of work — no AES-GCM credential storage, no KEK env var, no `ExchangeGrantCode` RPC, no kek-version rotation — because ADT can derive everything it needs from the per-request JWT. |
| B. Popup with `postMessage` callback | Same redirect URL but in a popup; the callback page calls `window.opener.postMessage(...)` and `window.close()`. | Rejected. Extend App UI runs inside the AGS Admin Portal iframe, so the popup's `window.opener` is the iframe — postMessage between cross-origin iframe and popup is browser-fragile; one browser update kills it. Popup blockers add a second failure surface. |
| C. Modal-only mock (no real redirect; admin types credentials) | Modal asks for ADT API key + namespace + gameId directly; no redirect at all. | Rejected. Requires the admin to copy-paste a long-lived API key from ADT — bad security posture, and ADT may not even expose long-lived keys for end-users in the service-JWT world. |

### D3 — Build URL: per-applicant, not per-playtest

| Option | Summary | Verdict |
| --- | --- | --- |
| **A. Per-applicant URL via ADT API at approve time** | Approve calls ADT (`POST /ext-playtesthub/.../invite` or whatever the eventual endpoint is — TBD-from-ADT-eng) with the applicant's identity, ADT returns a one-time / time-bounded download URL, the DM body embeds that URL. | **Chosen, contingent on ADT API exposing it.** Each applicant gets a distinct URL → ADT can audit/track per-tester downloads, revoke individuals without re-DM'ing everyone, bound URL lifetime. Aligns with how ADT already thinks about playtest membership. |
| B. Per-playtest URL configured at create time | Admin pastes a static "build download URL" into the create form; backend stores it on the playtest row; every approve DM embeds the same URL. | Rejected as primary; **kept as a fallback path** for the case where the resolved ADT API does not expose a per-applicant URL surface. The fallback is admin-visible (the create form shows the playtest-level static URL field as a "fill this if ADT can't issue per-applicant URLs" knob) and is one column on the Playtest row (`adtFallbackDownloadUrl TEXT?`). |
| C. Per-build URL the player resolves themselves | Player clicks a link, redirects to ADT, ADT auths the player against playtesthub's IAM. | Rejected. Adds a player-side ADT-auth round trip we can't validate against ADT today, and forecloses on the "player just gets a link in Discord and clicks" UX the PRD targets. |

### D4 — Auto-approve gating: synchronous in signup handler, single advisory lock per playtest

| Option | Summary | Verdict |
| --- | --- | --- |
| **A. Synchronous in `Signup` under `pg_advisory_xact_lock(hashtext('autoapprove:' || playtestId))`** | The signup tx takes the playtest-scoped advisory lock, counts existing `auto_approved=true` applicants, decides, and (if under cap) chains into the existing reserve→fenced-finalize→ApproveCAS path inside the same tx. | **Chosen.** Reuses every M2-vintage approve primitive verbatim; advisory lock is cheap and playtest-scoped so it doesn't serialize across the whole namespace. CAS keeps the post-lock concurrency safe even if the lock is somehow skipped. |
| B. Background worker that scans PENDING applicants and auto-approves | A new ticker job that walks `applicant.status='PENDING' AND playtest.auto_approve=true` and approves up to the cap. | Rejected. Adds latency between signup and DM (bad UX for the "instant access" feel auto-approve sells); adds another lease-elected worker; doesn't actually simplify the cap-enforcement logic. |
| C. Optimistic: just call approve from signup, swallow `ResourceExhausted` if cap hit | No advisory lock, no count. Race during the boundary can over-approve by `N=concurrent-signup-count`. | Rejected. Studios set `autoApproveLimit` precisely because they want a hard cap (limited Steam keys; limited build seats); over-shooting by even one is a real bug for them. |

**Cap semantics**: `autoApproveLimit` bounds **auto-approvals only**. Manual `ApproveApplicant` against PENDING applicants is unaffected and uncapped — admins can manually approve beyond `autoApproveLimit` if they choose. The applicant carries a new `auto_approved BOOLEAN NOT NULL DEFAULT FALSE` column so the count is unambiguous.

**Interaction with pool**: for STEAM_KEYS / AGS_CAMPAIGN, auto-approve still consumes a code from the pool — so the effective cap is `min(autoApproveLimit, pool_remaining)`. When the pool empties first, subsequent auto-approve attempts surface `ResourceExhausted` to the signup handler, which **falls back to PENDING** (signup still succeeds with `status=PENDING`; the auto-approve attempt is best-effort). No applicant-visible error.

| Option | Summary | Verdict |

### D6 — Distribution tab: per-model rendering, shared empty/configured-state scaffold

| Option | Summary | Verdict |
| --- | --- | --- |
| **A. One tab, per-model renderer (ADT / Steam / AGS)** | Distribution tab dispatches on `playtest.distributionModel`. ADT renders B7's link-state UI (unlinked empty state + Link ADT Namespace CTA; linked state shows namespace + build picker). ### D8 — Discord Bot Tools tab: bulk announcement DM + persistent history

| Option | Summary | Verdict |
| --- | --- | --- |

## Phase tracker

Phase status legend: `[ ]` not started · `[~]` in progress · `[x]` shipped.

Each shipped phase carries tests + smoke-harness extension + verification per the TDD + smoke rule in [`engineering.md`](engineering.md) §4–5.

Phases are split into two independent tracks. **Track A is priority 1 and lands before any Track B phase begins** — auto-approve has zero ADT dependency, ships value to operators today, and de-risks the milestone while ADT-eng's API answers settle.

### Track A — Auto-approve (priority 1, ships first)

A1. `[x]` **PRD §5.4 auto-approve subsection + [`schema.md`](schema.md) note + [`errors.md`](errors.md) rows + [`CHANGELOG.md`](CHANGELOG.md) v2.4** —

**Addendum 2026-05-21 — live-API probe corrections (against develop.blackbox.accelbyte.io v1.35.0, public swagger at `/profiling/apidocs/api.json`):**

Six bugs in the live `HTTPClient` adapter were discovered by direct probing today. All fixes shipped in commits `81da056`, `21b06bc`, `219558b`, `ac37ada` against `pkg/adt/http.go` + `pkg/adt/errors.go` + `pkg/adt/retry.go` + `pkg/service/adt.go`. The canonical reference for ADT shapes is now the live swagger URL above — supersedes the earlier `/home/ab/Downloads/adt-spec.txt` drop.

- **Bug 1 — `ListBuilds` envelope was wrong.** Old decoder expected `{"builds": [...]}`; live API returns `{"data": [...]}`. Silent zero-row failure mode against real ADT.
- **Bug 2 — `ListGames` envelope was wrong** (same root cause, same shape: `{"data": [...]}`, not `{"games": [...]}`).
- **Bug 3 — `IssueDownloadURL` envelope was wrong.** Old decoder expected `{"download_urls": [{url, expires_at}, ...]}`; live API returns `{"urls": [string, ...], "expires_at": "RFC3339"}` — flat string list with a single top-level expiry. The "first URL wins" semantic preserved; `IssuedDownloadURL.ExpiresAt` now sources the top-level expiry.
- **Bug 4 — Error classification was wrong.** Old code mapped HTTP 401 → `ErrLinkageMissing`. Live API uses a JSON `{errorCode, errorMessage}` envelope whose value is decoupled from the HTTP status:
  - `HTTP 404 + errorCode=99` ("Namespace is not registered") = real linkage-missing → `ErrLinkageMissing`.
  - `HTTP 401 + errorCode=401` ("unauthorized") = bearer broken → new `ErrUnauthenticated` (operator rotates `AGS_IAM_CLIENT_*`).
  - `HTTP 401 + errorCode=20001` ("unauthorized access") = token valid, route perm absent → new `ErrPermissionDenied` (ADT-eng grant request).
  - Helpers: `IsUnauthenticated(err)`, `IsPermissionDenied(err)` mirror existing `IsLinkageMissing`. Service-layer `recordUnlinkADTSideFailure` adds matching reason labels to `adt_unlink_adt_side_failures_total`. `RecoverADTLinkage` surfaces distinct `FailedPrecondition` messages so operators see the right action.
- **Bug 5 — Content-Type-aware error parsing.** ADT returns two body kinds on errors: JSON (`application/json`) carries the `errorCode` envelope; plaintext (`text/plain`) carries mux-level routing misses like `"404: Page Not Found"`. `doJSON` now branches on Content-Type before attempting `json.Unmarshal`; plaintext bodies fall through to HTTP-status classification without a spurious parse error.
- **Bug 6 — `Authorization: Bearer …` canary.** Live ADT rejects lowercase `bearer`. Production already uses capital `B`; the canary test pins the literal so a future refactor cannot silently downcase it. No production-code change for Bug 6.

## Resolved decisions (2026-05-19 scoping interview, refined 2026-05-19 follow-up)

1. **Linking scope → per-studio (upgraded from per-game-namespace).** D1. Linkage rows live keyed by `(studio_namespace, adt_namespace)`; reusable across every game namespace and every playtest under the studio. `studio_namespace` is derived server-side from the **playtesthub backend's own AGS service IAM JWT** as `union_namespace ?? namespace` (game-namespace service tokens always carry `union_namespace`; studio-namespace service tokens carry `namespace` equal to the studio namespace) — NOT from the calling admin's request token, since the service JWT is what ADT validates on every downstream API call. Rationale for the scope upgrade from the original "per-namespace": studios with multiple games would otherwise repeat the same operator dance per game with no safety gain — ADT validates the per-call service JWT and reads studio identity from its claims, so collapsing N game-scoped links into one studio-scoped link loses nothing.
2. **Linking signal → full-page redirect with `state` token + `studio_namespace` query param; NO `grantCode` exchange; `adt_namespace` echoed on redirect-back URL.** D2. Resolved the user's open question ("how does playtesthub know the link succeeded?") — admin clicks Proceed, admin UI persists form draft + redirects to `${ADT_BASE_URL}/oauth/link?state=…&redirect_uri=…&studio_namespace=…`, ADT records the `(adt_namespace, studio_namespace)` flag on its side and redirects back with `state=…&result=success&adt_namespace=<picked>` (no credential, no token, no `grantCode`). Admin UI calls `CompleteADTLink(state, adt_namespace)`, backend validates the `adt_link_pending` row and inserts the `adt_linkage` identity row. URL echo of `adt_namespace` is safe because (i) `state` defends CSRF and (ii) operator-browser tampering with `adt_namespace` is self-defeating — the first downstream ADT API call 401s loudly because ADT's flag for the tampered value is missing. Every subsequent ADT API call is authed by playtesthub minting an AGS service IAM JWT and sending `Authorization: Bearer …`; ADT validates against AGS JWKS and reads studio identity from `iss` / `union_namespace`. Consequences: no credential storage, no `ADT_CREDENTIAL_KEK` env var, no KEK rotation, no `ExchangeGrantCode` RPC, no encrypted ciphertext column on `adt_linkage`. Linking is **decoupled from playtest creation** because per-studio linkage is reusable across playtests — the user's "separate the linking process" instinct generalises to "do it on the studio, not on a specific playtest".
3. **Build URL strategy → per-applicant with static-fallback.** D3. ADT-issued URL preferred for revocability/audit; static `adtFallbackDownloadUrl` covers the case where ADT doesn't expose the per-applicant surface in time.
4. **Auto-approve gate → synchronous in `Signup` under playtest-scoped advisory lock.** D4. Reuses M2's approve primitives verbatim; cap applies to auto-approvals only; manual `ApproveApplicant` stays uncapped.
5. **Auto-approve availability → all three distribution models, not ADT-only.** User explicitly called this out — `autoApprove` + `autoApproveLimit` are model-agnostic playtest fields (the cap is on applicants, not codes/builds).
6. **Pool-empty during auto-approve → silent PENDING fallback.** Auto-approve is best-effort; the signup itself still returns success. Operator restocks the pool (STEAM_KEYS/AGS_CAMPAIGN) and `RetryFailedDms` is not the right mechanism (those applicants aren't `dm_failed`, they're `PENDING`); they just get manually approved or wait for the next auto-approve burst.
7. **Priority split — Track A (auto-approve) ships before Track B (ADT) begins.** Auto-approve has zero ADT dependency, reuses M2's approve primitives, and unblocks the daily-iteration-playtest pain point on STEAM_KEYS / AGS_CAMPAIGN today. Track B blocks **only** on the live-adapter sub-phase (Open questions §1–2 — per-applicant URL surface + build browse surface). Operationally: A1–A7 shipped → `main` CI green → B1 begins. The phase IDs reflect this (Ax / Bx instead of monotonic 1–18) so the tracker is unambiguous about which work block is in flight.
8. **ADT auth → AGS service IAM JWT, not a separately-issued ADT credential.** Follow-up resolution. Playtesthub mints a service JWT via the existing `pkg/ags` token getter (client_credentials grant against AGS IAM using `AGS_IAM_CLIENT_ID` / `AGS_IAM_CLIENT_SECRET`) and sends it as `Authorization: Bearer …` on every ADT call. ADT validates against the AGS IAM JWKS endpoint and reads studio identity from the token's `iss` URL + `union_namespace` claim — it does not need to remember anything other than the `(adt_namespace, studio_namespace) linked = true` flag set at link time. Eliminates the entire credential-storage + KEK-rotation surface the earlier draft assumed.
9. **`studio_namespace` is derived from the backend's service IAM JWT, NOT the calling admin's request token.** Correction landing alongside B4 (the implementation was right; the B1 / D1 / D2 / Resolved §1 / PRD §4.8.1 / errors.md / schema.md prose was wrong). Rationale: every downstream ADT API call from playtesthub carries the backend service JWT, so ADT's `(adt_namespace, studio_namespace) linked = true` flag is keyed on the *service token's* studio identity. Keying the playtesthub-side `adt_linkage` row on the admin's request-token claims would cause a flag mismatch any time the two tokens disagree (e.g. an admin token at game-namespace scope vs a service token at studio scope), surfacing as `IssueDownloadURL` 401s post-link. Fix-forward: PRD §4.8.1, schema.md §"adt_linkage table", errors.md `StartADTLink` row, and every D1 / D2 / Resolved §1 prose now reads "backend's service IAM JWT" instead of "calling admin's token". CHANGELOG bumped to v2.5.1 with the correction noted.
10. **Track C scope → UX shell + bulk announcements + new playtest metadata; per-applicant ADT telemetry fully deferred to M6.** Scoping decision 2026-05-20 from the UX prototype walkthrough at [`playtesthub-adt.netlify.app`](https://playtesthub-adt.netlify.app/), refined the same day after the user confirmed ADT telemetry will land in M6. Track C ships: the detail-page-with-tabs admin shell (D5), per-model Distribution tab (D6), 6-column Participants tab with derived Code Sent Date (D7, §16), Discord Bot Tools bulk announcements (D8), publish/stop verb rename (D9), banner + description + platforms playtest fields, and the breaking `platforms` column replacement (§15). Track C does **not** ship: the `pkg/adt.GetBuildTelemetry` client, the telemetry refresh worker, per-applicant Download Date / Total Playtime columns, the Hardware Specs + Crash Reports modal sections, or the "View in ADT" deep-link. Migration 0007 still lands the four nullable applicant cache columns + the dormant `applicant_adt_telemetry_stale_idx` so M6's worker + endpoint hookup is zero-schema-churn — they ship dormant in M5.C. Rationale: the prototype's UX rhythm is recognisable even without the telemetry payload (operators get the new shell, new tabs, new announcement broadcaster, new playtest fields) and M6 is already scoped against ADT-eng's telemetry endpoint timeline; bundling telemetry into M5.C would gate the whole UX revamp on an external dependency we don't control.
11. **Track C cap-reached behaviour → preserve M5.A silent PENDING fallback.** Scoping decision 2026-05-20. The prototype's Create-form subcopy ("Players who sign up after the limit is reached will see a 'playtest is full' message.") is treated as a mock inaccuracy. Rationale: the M5.A silent-PENDING fallback is a documented contract operators rely on for daily-iteration playtests (cap restocking is the operator workflow, not a player-visible error); rewriting the contract for cosmetic alignment would force a sweep of M5.A's concurrency tests, PRD §5.4 prose, and the documented operator runbook for cap restocking. C9's Create-form subcopy under `Max Participants` is reworded to `"Players who sign up after the limit is reached enter the manual approval queue."` — restoring the prototype's intent without changing M5.A behaviour.
12. **Track C distribution-tab rendering → per-model branches under a shared empty-state scaffold.** Scoping decision 2026-05-20 from the user's direction "infer the changes to other distribution's design based on the ADT distribution, the UI should be similar." The prototype only models ADT; STEAM_KEYS + AGS_CAMPAIGN inherit the same visual rhythm (empty-state card → CTA → populated state) with model-specific populated-state contents (code-pool table for Steam, campaign card for AGS, namespace+build picker for ADT). See D6.
13. **Track C admin shell → list + detail-page-with-tabs, NOT list + modal.** D5. The legacy modal-per-action shape is unviable for the four-tab detail surface; per-tab URL permalinks (`/playtest/<slug>?tab=participants`) are a load-bearing operator workflow.
14. **Track C blocks on Track B.** Cross-cutting rule. The Distribution tab reuses B4's linkage state machine. Operationally: B1–B11 shipped → `main` CI green → C1 begins.
15. **Track C `platforms` migration → drop the old single-platform column, NOT backfill the array.** Scoping decision 2026-05-20. The two candidates considered: (a) backfill `platforms = ARRAY[existing_platform_column]`, retain the old column for a deprecation cycle; (b) **chosen** — drop the old column in migration 0007, accept that every existing playtest reads `platforms = '{}'` post-deploy until an operator Edits it. Rationale: keeps the schema strict (no two-source-of-truth window during the deprecation cycle); operator UX regression is bounded (player Landing renders "Platforms: not specified" until Edit); pre/post-migration runbook step in `runbooks/admin-shell-tour.md` (C12) documents the manual re-pick. Migration unit test asserts the old column is gone (regression canary against accidental re-add). **Breaking change to the schema** — called out explicitly in CHANGELOG v2.6.
16. **Track C `Code Sent Date` for Steam/AGS participants → derive at read time from `dm.sent` audit row, NOT a new applicant column.** Scoping decision 2026-05-20. The participants table needs a "when did this applicant receive their code DM" timestamp for STEAM_KEYS / AGS_CAMPAIGN rows; the M2 `dm.sent` audit row already carries this exact timestamp. Adding a denormalized `applicant.code_sent_at` column would duplicate state with no read-perf gain (one LATERAL join against `audit_log` filtered on `action='dm.sent'` resolves to an index hit). The derived path keeps the applicant schema lean and side-steps a backfill question for existing rows. ADT-distribution applicants don't have a Code Sent Date in M5.C (analogue is the deferred-to-M6 Download Date).
